# cog-verifier — Субагент-верификатор

## Назначение

Автономное выполнение пресетов верификации. Отделяет контекст
верификации от основного агента, чтобы не засорять рабочий контекст
и обеспечить независимую проверку.

## Приоритет реализации

### Путь А: делегирование в `verification` plugin

Если установлен плагин `verification:verify`:
1. Вызвать `Skill: "verification:verify"` с аргументами: `preset`, `target`,
   `scope`.
2. Получить результат: passed / failed / warnings.
3. Проброс результата вызывающему скиллу.

### Путь Б: внутренний fallback

Если плагин недоступен — применить упрощённый пресет из
`references/verification-fallbacks.md`:

- Для П5 «Максимум»: упрощённый до 10 ключевых механизмов.
- Для П13 «Аудитор»: 5 механизмов (М18, М13, М28, М10, М11).
- Для П14 «Итератор»: 4 механизма (М25, М26, М24, М22, М32).
- Для П12 «Стратег»: 6 механизмов (М16, М17, М10, М27, М28, М12, М14).

## Поддерживаемые пресеты

| Пресет | Когда вызывается | Кем |
|---|---|---|
| П5 «Максимум» | Критичные находки аудита, крупные миграции | [[audit]], [[migrate]] |
| П12 «Стратег» | wm-NNN, md-NNN, долгосрочные решения | [[patterns]] (промоушен) |
| П13 «Аудитор» | Любой [[audit]], крупные сверки | [[audit]] |
| П14 «Итератор» | После любых правок в продовых карточках | [[archive]], [[migrate]], [[audit]] |
| П7 «Спикер» | Перед финальным обсуждением с пользователем | [[reflect]] monthly |
| М1 (базовый) | Мелкие правки | все |

## Входные параметры

```yaml
workspace: /path/to/cognitive_os
preset: P5 | P12 | P13 | P14 | M1 | ...
target:
  type: audit_record | migration_result | archival_result | pattern | ...
  ids: [audit-018, audit-019]
  files: [14_audit_log.md, 02_patterns.md]
scope: full | diff_only | ids_only
caller: audit | migrate | archive | ...
timeout_seconds: 300
```

## Workflow

### 1. Проверка доступности плагина

```bash
# Проверяем наличие verification:verify
```

Если доступен — путь А. Иначе — путь Б.

### 2. Загрузка target

- Если `files` указаны — прочитать все.
- Если `ids` указаны — найти их во всех `.md`.
- Если `scope=diff_only` — взять `git diff` или снимок из
  `archive/.backup/`.

### 3. Прогон механизмов

Для каждого механизма пресета:
- Выполнить соответствующую проверку.
- Собрать найденные проблемы (с severity).
- Собрать подтверждения (что проверено успешно).

### 4. Генерация отчёта верификации

```markdown
## Результаты верификации — {{DATE}}

**Пресет:** {{PRESET}}
**Target:** {{TARGET}}
**Caller:** {{CALLER}}

### Что проверено

- М1 (точный перенос) — ✅ 42 цифры, 15 дат, 8 имён совпадают.
- М25 (регрессия) — ✅ нет неожиданных удалений.
- М26 (дрейф смысла) — ⚠ обнаружен возможный дрейф в km-012.

### Что найдено

- **severity: high** — М26: определение «ФП» изменилось с v1 на v2:
  было «8 федеральных проектов», стало «7». Требуется разъяснение.
- **severity: low** — М32: one orphan wikilink `[[md-099]]`.

### Уверенность

0.85 (высокая)

### Рекомендации

1. Создать audit-NNN для разбора км-012.
2. Добавить [[md-099]] или удалить ссылку.
```

### 5. Возврат результата вызывающему скиллу

```yaml
status: passed | failed | passed_with_warnings
issues:
  - id: issue-001
    mechanism: M26
    severity: high
    description: "Дрейф смысла в km-012"
    suggested_action: "Создать audit-NNN"
confirmations:
  - mechanism: M1
    result: "Все цифры, даты, имена совпадают"
confidence: 0.85
full_report_path: <workspace>/docs/verification/{{DATE}}_{{CALLER}}.md
```

### 6. Запись отчёта

Сохранить в `<workspace>/docs/verification/{{DATE}}_{{CALLER}}_{{PRESET}}.md`
для истории.

## Механизмы во внутреннем fallback

### П14 «Итератор» — упрощённая версия

- **М25 регрессия:** сравнить diff, найти удаления без обоснования.
- **М26 дрейф смысла:** сравнить определения ключевых терминов v1 vs v2.
- **М24 diff было/стало:** сгенерировать и проверить разумность.
- **М22 согласованность файлов:** counters, cross-references.
- **М32 ссылочная целостность:** все wikilinks resolve.

### П13 «Аудитор» — упрощённая версия

- **М18 триангуляция источников:** проверить упоминания в ≥ 2 карточках.
- **М13 декомпозиция фактов:** разбить утверждение на атомы, проверить
  каждый.
- **М28 эпистемический статус:** проверить метки (факт / мнение /
  гипотеза).
- **М10 скрытые допущения:** выявить неявно принятое.
- **М11 парадоксы:** поиск внутренних противоречий.

### П5 «Максимум» — упрощённая версия

Ключевые 10 механизмов: М1, М6, М7, М10, М13, М14, М18, М22, М25, М27.

## Интеграции

- `verification` plugin — основной путь.
- `references/verification-fallbacks.md` — fallback.
- [[audit]], [[calibrate]], [[reflect]], [[archive]], [[migrate]],
  [[patterns]] — вызывающие скиллы.
- `docs/verification/` — архив отчётов.
- [[14_audit_log]] — здесь могут создаваться записи на основе находок.

## Производительность

| Пресет | Target size | Время (плагин) | Время (fallback) |
|---|---|---|---|
| М1 | 1 файл | < 1 сек | < 1 сек |
| П14 | 10 файлов | 5-15 сек | 3-8 сек |
| П13 | вся 08 | 15-30 сек | 8-20 сек |
| П5 | крупный diff | 30-90 сек | 20-60 сек |

`timeout_seconds` — hard-limit. При превышении — частичный отчёт и
предупреждение.

## Частые ошибки

1. **Рекурсивный вызов** (скилл вызывает верификатор, тот снова скилл).
   Защита: флаг `in_verification=true` в аргументах.
2. **Недоступен verification plugin** — fallback обязателен, иначе
   пропуск верификации = silent bug.
3. **Слишком много findings** — сортировать по severity, отчёт
   обрезать до top-20 (остальное в full_report_path).
