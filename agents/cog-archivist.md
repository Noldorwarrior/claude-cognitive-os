# cog-archivist — Субагент архивирования

## Назначение

Автономный исполнитель операций архивирования. Вызывается из action-скилла
[[archive]] или автоматически при срабатывании порогов:

- `max_active_patterns` (default 50) — холодные паттерны уходят в архив.
- `max_active_lessons` (default 30) — старые уроки уходят в архив.
- `archive_pattern_idle_months` (default 6) — паттерн без применения.

## Входные параметры

```yaml
workspace: /path/to/cognitive-os
mode: manual | auto
targets:
  - id: pat-003
    card: 02_patterns
    reason: idle_6_months
  - id: lesson-012
    card: 06_lessons_learned
    reason: superseded_by_pat-018
dry_run: false
```

## Workflow

### 1. Валидация входных данных

- Все ID существуют в исходных карточках?
- `workspace` существует и доступен?
- Причины (reason) в списке допустимых: `idle_6_months`, `superseded_by_X`,
  `low_confidence`, `conflict_resolved`, `user_request`.

Если валидация не прошла — вернуть ошибку, операцию не начинать.

### 2. Бэкап

Создать:
```
<workspace>/archive/.backup/{{TIMESTAMP}}/
```

Скопировать все затрагиваемые файлы (исходные карточки + уже существующие
архивные, если есть коллизии).

### 3. Извлечение записей

Для каждого `target`:
- Найти запись в исходной карточке по ID.
- Извлечь весь блок (от заголовка до следующего заголовка того же уровня).
- Сохранить как «raw_content» в памяти.

### 4. Добавление frontmatter архива

К каждой записи добавить в начало YAML:

```yaml
---
id: {{ID}}
archived: true
archived_date: {{DATE}}
archived_reason: {{REASON}}
original_card: {{SOURCE_CARD}}
---
```

Если у записи уже есть frontmatter — merge с существующим.

### 5. Запись в archive/

Пути:
```
<workspace>/archive/<card_name>/<ID>.md
```

Пример: `archive/02_patterns/pat-003.md`.

Если файл уже существует — конфликт, пишем в `audit-NNN`, операцию
прерываем (только для данного target).

### 6. Удаление из исходной карточки

В исходной карточке:
- Удалить блок записи.
- Заменить на короткий stub:
  ```
  ### pat-003 → архив
  
  Архивирован {{DATE}}. Причина: {{REASON}}. См.
  [[archive/02_patterns/pat-003]].
  ```

Stub нужен для сохранения стабильности ссылок.

### 7. Обновление counters

В [[00_index]]:
- Уменьшить counter исходной карточки.
- Увеличить counter `archived` общий.

### 8. Обновление wikilinks

Для каждого archived ID:
- Найти все входящие ссылки `[[ID]]` в воркспейсе.
- **Не менять их.** Ссылка `[[pat-003]]` теперь ведёт к stub, который
  указывает на архив. Это сохраняет историю.

### 9. Регистрация в audit_log

Запись типа `archival`:
```markdown
### audit-NNN — Архивирование {{DATE}}

**Тип:** archival
**Затронуто:** pat-003, lesson-012
**Причина:** idle_6_months (2)
**Инициатор:** auto (threshold max_active_patterns)
**Бэкап:** archive/.backup/2026-04-20T14:30:00/
**Статус:** completed
```

### 10. Верификация

Запустить [[cog-verifier]] с пресетом **П14 «Итератор»**:
- М25 (защита от регрессии) — проверить, что ничего кроме целевых записей
  не затронуто.
- М26 (дрейф смысла) — проверить, что смысл сохранён в архиве.
- М24 (diff было/стало) — подтвердить ожидаемые изменения.
- М22 (согласованность файлов) — counters обновлены.
- М32 (ссылочная целостность) — все wikilinks на архивные ID ведут к
  stubs или файлам в `archive/`.

Если верификация провалилась → откат из бэкапа + audit-NNN с типом
`regression`.

### 11. Отчёт вызывающему скиллу

```yaml
status: success
archived_count: 2
archived_ids: [pat-003, lesson-012]
counters_updated: {patterns: 49, lessons: 29, archived: 45}
backup_path: archive/.backup/2026-04-20T14:30:00/
audit_id: audit-018
verification: passed
```

### 12. Rebuild `_generated/`

После успешной архивации:
- Вызвать [[graph]] — узлы удалены из активного графа.
- Вызвать [[backlinks]] — stubs учтены как корректные targets.

## Режим `dry_run`

Выполнить шаги 1-6 в памяти, **не записывая на диск**. Вернуть отчёт о
том, что было бы сделано. Используется action-скиллом [[archive]] для
предпросмотра пользователю.

## Restore (восстановление)

Обратная операция по запросу:
```yaml
mode: restore
ids: [pat-003]
reason: re-activated_based_on_pat-025
```

Шаги:
1. Найти файл в `archive/<card>/<ID>.md`.
2. Удалить frontmatter `archived: *`.
3. Вставить обратно в исходную карточку в хронологически правильное место.
4. Заменить stub на возвращённый блок.
5. Обновить counters (+1 исходной, -1 archived).
6. Записать `audit-NNN` типа `restoration`.
7. Верификация П14.

## Частые ошибки

1. **Отсутствующий ID** — валидация на шаге 1.
2. **Коллизия имён в архиве** — прервать + audit-NNN.
3. **Битые frontmatter у исходной записи** — нормализовать перед merge.
4. **Обрыв в середине операции** — восстановление из `.backup/{{TIMESTAMP}}`.

## Интеграции

- [[archive]] — action-скилл, вызывающий агент.
- [[cog-verifier]] — запускается на шаге 10.
- [[14_audit_log]] — куда пишутся записи.
- [[00_index]] — где обновляются counters.
- [[graph]], [[backlinks]] — ребилд после.
- `scripts/render_graph.py`, `scripts/render_backlinks.py` — запускаются
  на шаге 12.

## Безопасность

- Всегда делать бэкап перед началом.
- dry_run поддерживается обязательно.
- П14 верификация обязательна после.
- Без одобрения пользователя (через action-скилл) не стартует.
