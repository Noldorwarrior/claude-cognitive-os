# cog-detector — Фоновый субагент-детектор

## Назначение

Непрерывный анализ воркспейса на предмет сигналов, требующих действий:

- **Паттерн-кандидат:** 3+ однотипных события в окне
  `pattern_window_days`.
- **Кластер-кандидат:** 3+ проекта с общим доменом.
- **Systemic-ошибка:** 3+ экземпляра одной ошибки в разных контекстах.
- **Потенциальный конфликт:** два близких по теме knowledge-map с
  противоречивыми утверждениями.
- **Пороги у срабатывания:** counter ≥ 80% от threshold.
- **Холодные записи:** паттерны/уроки без применения > 80% от
  `archive_pattern_idle_months`.

## Режимы запуска

### По hook (автоматический)

`hooks/hooks.json` → `post-write` на `projects/**/*.md` и `0[0-9]_*.md`:
```bash
python3 ${CLAUDE_PLUGIN_ROOT}/scripts/run_detector.py \
  --workspace ${CLAUDE_WORKSPACE} \
  --changed "${CLAUDE_CHANGED_FILES}"
```

Режим инкрементальный: анализируем только затронутые карточки + их
соседей по графу.

### По расписанию (cron / scheduled task)

Раз в сутки полная проверка всего воркспейса.

### По требованию

Команда «что детектор видит» → полный отчёт.

## Сигналы и их обработка

### 1. Паттерн-кандидат

**Триггер:** 3+ однотипных событий (`pattern_candidate` в observations)
с `similarity ≥ 0.75` в `pattern_window_days`.

**Действие:** записать в буфер `_generated/detector_signals.md`:
```markdown
### Паттерн-кандидат (2026-04-20)

**Тип:** pattern_candidate
**Совпадений:** 3
**Окно:** 2026-03-21 — 2026-04-20 (30 дней)
**Источники:**
- 02_patterns#observation-042 (2026-03-25)
- 02_patterns#observation-051 (2026-04-10)
- 02_patterns#observation-058 (2026-04-18)
**Сходство:** 0.82
**Предложение:** создать pat-NNN через [[patterns]].
```

При следующем взаимодействии основной агент увидит сигнал и предложит
пользователю: «Детектор обнаружил паттерн-кандидат. Запустить
[[patterns]]?»

### 2. Кластер-кандидат

**Триггер:** 3+ проекта в [[03_projects_registry]] с общим
`primary_domain` (или 2+ пересечением в `entities`).

**Действие:** сигнал `cluster_candidate` с предложением запустить
[[patterns]] в режиме cluster-сборки.

### 3. Systemic-ошибка

**Триггер:** 3+ записей в [[10_error_corrections]] с близким
`error_type` в разных `project_id`.

**Действие:** сигнал `systemic_error`, предложение запустить
[[calibrate]] для домена.

### 4. Потенциальный конфликт

**Триггер:** пара `(km-A, km-B)` с overlap ключевых терминов ≥ 60% и
различием в бинарных утверждениях (числа / «централизованный» vs
«децентрализованный» / имена).

**Действие:** сигнал `conflict_suspect`, предложение запустить [[audit]]
в режиме `--contradictions-only`.

### 5. Порог у срабатывания

**Триггер:** любой counter ≥ 80% от threshold.

**Действие:** в sidebar-уведомлении через [[status]]:
```
⚠ max_active_patterns: 48 / 50 (96%)
  Рекомендация: [[archive]] или [[patterns]]→промоушен в wm-NNN.
```

### 6. Холодные записи

**Триггер:** паттерн без `last_applied` за последние
`archive_pattern_idle_months * 0.8` месяцев (т.е. 4.8 месяца при default 6).

**Действие:** сигнал `cold_candidate` для [[archive]].

## Где пишет результаты

**Ничего не пишет в продовые карточки.** Только в:

- `<workspace>/_generated/detector_signals.md` — буфер сигналов.
- `<workspace>/_generated/detector_stats.json` — статистика для
  [[status]].
- `<workspace>/_generated/thresholds_near_trigger.json` — для sidebar.

Основной агент ЧИТАЕТ этот буфер при каждом старте новой задачи и при
запросе [[status]].

## Инкрементальный режим

При вызове с `--changed "fileA.md,fileB.md"`:
1. Загрузить только affected файлы.
2. Найти их ID и соседей по backlinks.
3. Перепроверить сигналы только в этом подмножестве.
4. Merge с существующим `detector_signals.md` (старые сигналы
   сохраняются, новые добавляются).

## Производительность

| Размер воркспейса | Полный проход | Инкрементальный |
|---|---|---|
| < 100 записей | < 1 сек | < 0.5 сек |
| 100-500 | 1-5 сек | < 1 сек |
| 500-2000 | 5-20 сек | 1-3 сек |
| > 2000 | используй агент-планировщик | 3-5 сек |

Hook всегда инкрементальный. Полный — по расписанию или вручную.

## Отладка

```bash
# Показать все сигналы, что детектор видит
python3 scripts/run_detector.py --workspace . --verbose --stdout

# Проверить конкретный тип
python3 scripts/run_detector.py --workspace . --only conflict_suspect
```

## Что детектор НЕ делает

- Не модифицирует карточки.
- Не создаёт pat-NNN / wm-NNN / audit-NNN напрямую.
- Не решает, нужно ли действие — только предлагает.
- Не переписывает backlinks / graph.

Все финальные действия — через action-скиллы при взаимодействии с
пользователем.

## Интеграции

- [[patterns]] — потребитель сигналов `pattern_candidate`,
  `cluster_candidate`.
- [[calibrate]] — потребитель `systemic_error`.
- [[audit]] — потребитель `conflict_suspect`.
- [[archive]] — потребитель `cold_candidate`.
- [[status]] — показывает `thresholds_near_trigger`.
- [[cognitive-os-core]] — источник правил / порогов.
- `scripts/run_detector.py` — сам исполнитель.
- `hooks/hooks.json` — триггеры.

## Частые ошибки

1. **Ложные кластер-кандидаты** (домены похожи, но суть разная).
   Mitigation: detector предлагает, пользователь решает.
2. **Размножение сигналов** при массовых правках. Mitigation:
   debounce в hooks + дедупликация в detector_signals.md по
   `(type, source_ids)`.
3. **Засорение буфера** устаревшими сигналами. Mitigation: TTL — сигналы
   старше 14 дней автоматически удаляются при следующем проходе.
