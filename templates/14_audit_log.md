# 14 — Журнал расхождений между проектами / картами

Фиксация **противоречий и расхождений** между записями в разных
картах или между проектами. Заполняется при запуске action-скилла
`audit` и через триггеры от субагента [[cog-verifier]].

**Что попадает сюда:**
- Противоречие между двумя картами знаний.
- Противоречие между механизмом и паттерном.
- Противоречие между проектами (один заявляет X, другой Y).
- Рассинхрон порогов между файлами (через `sync_check.py`).
- Несоответствие между версиями карты после правки (регрессия).

**Что НЕ попадает:**
- Ошибки Claude (→ [[10_error_corrections]]).
- Уроки (→ [[06_lessons_learned]]).
- Обычные правки / эволюция (→ в эволюцию карты).

## Типы расхождений

### knowledge_conflict
Противоречие в фактах между двумя картами знаний.
- Пример: km-001 утверждает «8 ФП нацпроекта», km-005 говорит «7 ФП».
- Разрешение: триангуляция источников, обновление устаревшей.

### pattern_conflict
Паттерн противоречит другому паттерну или мета-решению.
- Пример: pat-007 «SWOT first», pat-012 «Backcasting first».
- Разрешение: уточнение контекстов применения (возможно, оба валидны
  в разных случаях).

### project_conflict
Проекты декларируют противоречивые цели / подходы.
- Пример: proj-001 «централизация», proj-003 «децентрализация».
- Разрешение: вынесение в [[12_cross_project_graph]] как
  conflict-edge, обсуждение с пользователем.

### threshold_mismatch
Пороги в разных файлах рассинхронизированы.
- Пример: SKILL.md::thresholds.pattern=3, 02_patterns.md::thresholds.pattern=5.
- Разрешение: `sync_check.py` блокирует запись до синхронизации.

### regression
После итеративной правки что-то потерялось.
- Пример: в v1 было 3 раздела, в v2 осталось 2 без объяснения.
- Разрешение: М25 «защита от регрессии» → восстановить или
  зафиксировать намеренное удаление.

### scope_drift (v1.3)
Определение понятия сместилось между итерациями документа (М26 дрейф
смысла).
- Разрешение: вернуть исходное определение или явно зафиксировать
  переопределение.

## Жизненный цикл записи

```
open (обнаружено)
  ↓
under_review (назначена проверка)
  ↓
resolved | accepted_as_context_diff
```

- `resolved` — расхождение устранено.
- `accepted_as_context_diff` — оба варианта валидны в разных контекстах,
  фиксируется явно.

## Правила записи

1. **Уникальный ID** `audit-NNN`.
2. **Тип обязателен** из enum.
3. **Severity обязателен.**
4. **Источники** указаны с конкретными якорями.
5. **Суть расхождения** явно описана («A утверждает X, B утверждает Y»).
6. **Предложенное разрешение** — минимум один вариант.
7. **Решение пользователя** — фиксируется перед переводом в resolved.

## Связь с верификацией

- **Высокий severity** (high / critical) → автоматический вызов
  [[cog-verifier]] с пресетом П5 «Максимум».
- **Medium** → пресет П13 «Аудитор».
- **Low** → ручной чек-лист.

Результат верификации пишется в `docs/verification_log.md` и здесь в
записи.

## Автоаудиты

Запускаются регулярно через:
- `audit`-скилл — полный разбор на противоречия.
- `scripts/audit_contradictions.py` — быстрый скан (скрипт
  self-learning слоя).
- [[cog-verifier]] — точечная проверка после правок.

Периодичность автоаудита: раз в `thresholds.revision_interval_days`
(default 30) или по явному запросу.

## Команды

| Команда | Действие |
|---|---|
| `аудит` / `проверь противоречия` | Запустить `audit`-скилл |
| `покажи расхождения` | Все открытые |
| `разреши audit-X` | Применить выбранное разрешение |
| `acceptable context diff: audit-X` | Закрыть как валидное разделение |

## Связанные карты

- Все карты знаний — источники для аудита.
- [[10_error_corrections]] — если расхождение породило ошибку.
- [[12_cross_project_graph]] — conflict-edges между проектами.
- [[13_self_reflection]] — самоаудиты, выявляющие расхождения.
- [[verification]] plugin — пресеты проверки.
