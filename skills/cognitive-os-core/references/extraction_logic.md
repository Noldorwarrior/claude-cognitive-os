# Логика извлечения: паттерны / механизмы / ошибки

Это «движок» когнитивного слоя. Определяет **что**, **когда** и **куда**
записывать.

## Централизация порогов

Все значения «3+», «30 дней», «50 активных» — из `SKILL.md::thresholds`:

| Параметр | Default | Применение |
|---|---|---|
| `pattern` | 3 | Повторений до фиксации `pat-NNN` |
| `pattern_window_days` | 30 | Окно подсчёта повторений |
| `systemic_error` | 3 | Повторений ошибки → статус `systemic` |
| `knowledge_map_access` | 3 | Обращений к знанию → `km-NNN` |
| `max_active_patterns` | 50 | Лимит активных паттернов |
| `max_active_lessons` | 30 | Лимит активных уроков |
| `archive_entity_idle_months` | 6 | Без упоминаний → архив |
| `revision_interval_days` | 30 | Периодическая ревизия |
| `confidence_min_tasks` | 5 | Задач в домене → `11_confidence_scoring.md` |
| `cluster_min_projects` | 3 | Минимум проектов для кластера |

**Sync-правило:** при изменении значений в frontmatter `SKILL.md`
синхронно меняется поведение движка. Дублирующие значения в
`02_patterns.md::trigger_threshold` и
`10_error_corrections.md::systemic_threshold` должны быть приведены к
тем же числам. Автоматическая проверка — `scripts/sync_check.py`
(запускается перед любой записью в v1.3).

## Три потока наблюдений

| Поток | Источник | Куда | Триггер |
|---|---|---|---|
| Поведенческий | Реакции / предпочтения пользователя | `02_patterns.md` | `thresholds.pattern`+ повторений |
| Механический | Способы решения задач | `09_working_mechanisms.md` | Сразу при обнаружении |
| Ошибочный | Ошибки и их исправления | `10_error_corrections.md` | Сразу при обнаружении |

Потоки связаны: одна ошибка может породить и механизм, и паттерн.

## Поток 1: паттерны

### Алгоритм фиксации

```
1. После КАЖДОГО взаимодействия Claude мысленно отмечает:
   - какие реакции пользователя были
   - какие предпочтения проявились
   - какие повторяющиеся форматы запросов

2. Если наблюдение повторилось thresholds.pattern+ раз
   за thresholds.pattern_window_days:
   - grep по 02_patterns.md — может паттерн уже есть
   - Если нет — создать pat-NNN
   - Уведомить: «Заметил паттерн: [...]. Зафиксировал»

3. Если разовое или 2 раза — НЕ записывать.
   Счётчик ведётся только в текущем диалоге.
```

### Категории паттернов

| Категория | Пример |
|---|---|
| `behavior` | «после длинных ответов запрашивает короче» |
| `format` | «всегда хочет markdown с заголовками» |
| `methodology` | «всегда требует SWOT-анализ для стратегий» |
| `tool_usage` | «предпочитает файлы, не computer use» |
| `escalation` | «при ошибке — разбор причины, не извинения» |

### Что НЕ паттерн

- Один разовый эпизод (даже эмоциональный).
- Что-то уже в `user_preferences` (правила #1-#10) — не дублировать.
- Реакция на ошибку Claude (это поток 3).

## Поток 2: механизмы

### Алгоритм фиксации

```
1. Когда Claude:
   - удачно решает задачу нестандартным способом, или
   - адаптирует чужой скилл/правило к своей работе, или
   - изобретает новый workflow,
   и это решение масштабируемо на КЛАСС задач —

2. ЭТО МЕХАНИЗМ. Запись в 09_working_mechanisms.md:
   - wm-NNN
   - категория: workflow / format / analysis / communication / verification / tool_usage
   - триггер применения
   - шаги 1-2-3
   - что НЕ делать
   - связь с правилами / скиллами

3. Уведомить пользователя ТОЛЬКО ЕСЛИ механизм важный:
   «Зафиксировал механизм: [название]. Буду применять
   в задачах типа [...]»
```

### Источники механизмов

| Источник | Как обнаружить |
|---|---|
| `observation` | Сам разработал, сработало, понимаю почему |
| `error_correction` | Исправил ошибку → понял, как избегать |
| `skill_adoption` | Перенял из скилла (например, `govdoc-analytics`) |
| `user_rule` | Из правила пользователя |
| `user_correction` | Пользователь поправил → запомнить |

### Что НЕ механизм

- Однократный приём, не масштабируемый.
- Что-то уже описанное в скилле — лучше сослаться на скилл.
- Микро-приём уровня «запятая после однако».

## Поток 3: ошибки

### Алгоритм фиксации

```
1. Любая ошибка (свою заметил или пользователь указал)
   → ОБЯЗАТЕЛЬНАЯ запись.

2. Категория:
   - rule_violation
   - methodology_skip
   - format_error
   - data_error
   - logic_error
   - communication_error
   - tool_misuse

3. В записи:
   - что произошло (фактически)
   - корневая причина (анализ)
   - как исправлено
   - производный механизм (если формулируется)

4. Если эта же ошибка возникала thresholds.systemic_error+ раз
   → статус `systemic`:
   - Создать паттерн
   - Усилить механизм защиты
   - Возможно — мета-решение в 04_meta_decisions.md

5. Уведомить:
   «Зафиксировал ec-NNN. Извлёк wm-XXX.
   [N]-е повторение → статус systemic».
```

### Связь «ошибка → механизм защиты»

| Ошибка | Механизм защиты |
|---|---|
| Нарушил правило #6 (формат docx) | wm: «чек-лист правила #6 перед каждым docx» |
| Пропустил методологию (правило #8) | wm: «жёсткий блок: методология первой» |
| Перепутал имя организации | wm: «сверка с `05_global_glossary.md` + `08_knowledge_maps.md`» |
| Ошибка в расчётах | wm: «двойной расчёт + М2 верификации» |
| Длинный ответ на короткий вопрос | wm: «оценка контекста: разговор / документ» |
| Неправильный инструмент | wm: «дерево решения: правило → MCP → computer use» |

## Поток 4 (v1.3): карты знаний

### Алгоритм фиксации

```
1. Каждое обращение к предметному знанию (нацпроект, структура
   организации, справочник, regulation):
   - увеличить счётчик обращения в 08_knowledge_maps.md (внутренний)

2. Если счётчик достиг thresholds.knowledge_map_access:
   - создать km-NNN
   - заполнить: домен, ключевые узлы, источник, дата актуализации
   - уведомить: «Знание про X использовалось N+ раз, создаю km-NNN»

3. Пользователь может создать km принудительно командой
   `добавь знание про {домен}`.
```

## Режимы работы

### Режим 1. ИНИЦИАЛИЗАЦИЯ

Первый запуск плагина в папке:

1. Проверить: есть ли `CLAUDE.md` (если нет — создать).
2. Создать папку `cognitive_os/` (или использовать папку проекта, если
   `migrate`-режим).
3. Создать 11 базовых карт (00-10) из `templates/` плагина.
4. Заполнить `01_user_profile.md` из `user_preferences` (10 правил).
5. Заполнить `03_projects_registry.md` из существующей структуры
   `Projects/` (если есть).
6. Запустить `render_graph.py` / `render_backlinks.py` / `render_mermaid.py`
   первый раз.
7. Уведомить: «Когнитивный слой инициализирован. Создано N карт,
   собран граф на K узлов».

Action-скилл: `init`.

### Режим 2. РАБОТА (внутри обычной задачи)

1. Прочитать уровень 0-1 reading ladder (≈3 KB).
2. Если задача требует — поднять до уровня 2-3.
3. Решать задачу как обычно.
4. По ходу — фиксировать паттерны / механизмы / ошибки.
5. Post-write hook автоматически обновляет `_generated/graph.html` и
   `_generated/backlinks.md` (debounce 3-5 сек).
6. В конце задачи — обновить `00_index.md` и (если нужно) запустить
   самоаудит через action-скилл `reflect`.

### Режим 3. РЕВИЗИЯ (периодический пересмотр)

Каждые `thresholds.revision_interval_days` дней или по команде
`пересмотри глобальный слой` / action-скиллом `calibrate` + `audit`:

**Шаг 1. Делегирование `consolidate-memory` (если доступен):**
1. Объединить дубликаты в `02_patterns.md`.
2. Обновить устаревшие паттерны.
3. Перепроверить `08_knowledge_maps.md`.
4. Сверить `03_projects_registry.md` с реальной структурой `Projects/`.
5. Обновить `11_confidence_scoring.md` из самоаудитов.

**Шаг 2. Fallback (`fallbacks.consolidate_memory:
internal_minimal_revision`):**
1. **Дедупликация по grep:** для каждой карты — поиск одинаковых ID и
   первых 80 символов описаний → объединить в самую старую запись.
2. **Архивация:** записи без обновлений >
   `thresholds.archive_entity_idle_months` → в
   `_archive/{file}_{YYYY-MM}.md`.
3. **Обновление YAML:** `total`, `active`, `archived`, `updated`.
4. **Запись в `14_audit_log.md`:** короткая `revision-NNN` с датой,
   списком архивированных, режимом (`consolidate-memory` или
   `internal_minimal_revision`).
5. **Уведомление:** «Ревизия в fallback-режиме — результаты могут быть
   менее полными».

**Шаг 3 (v1.3): автокалибровка порогов.**
Запустить `scripts/calibrate_thresholds.py`. Скрипт анализирует
`11_confidence_scoring.md` и историю записей, предлагает новые пороги.
Пороги меняются только с явного согласия пользователя.

**Шаг 4 (v1.3): извлечение принципов.**
Запустить `scripts/extract_principles.py` — группирует уроки по топикам,
при 3+ похожих выводах формирует принцип для `04_meta_decisions.md`.

**Шаг 5 (v1.3): аудит противоречий.**
Запустить `scripts/audit_contradictions.py` — ищет противоречия между
записями, пишет их в `14_audit_log.md`.

Action-скилл: `audit` + `calibrate`.

## Цикл «наблюдение → запись → применение»

```
[Диалог]
   ↓
[Claude решает задачу]
   ↓
[Внутренняя проверка: что произошло?]
   ↓
   ├─ Поведение пользователя? → счётчик (3+ → pat-NNN)
   ├─ Новый способ работы? → wm-NNN сразу
   ├─ Ошибка? → ec-NNN сразу + извлечь wm-NNN
   ├─ Обращение к знанию? → счётчик (3+ → km-NNN)
   ↓
[sync_check.py проверяет пороги]
   ↓
[Запись в соответствующую карту]
   ↓
[Post-write hook: debounce 3-5 сек]
   ↓
[Пересборка graph.html / backlinks.md / graph.mermaid]
   ↓
[Обновление 00_index.md (статистика)]
   ↓
[Применение в следующих задачах]
```

## Защита от шума и инфляции

1. **Дедупликация:** grep по существующим ID и текстам перед каждой
   записью. Если похожее есть — обновить существующее.
2. **Обобщение:** 3+ паттернов про одно и то же → принцип в
   `06_lessons_learned.md`, остальные архивировать.
3. **Лимиты:**
   - Активных паттернов ≤ `thresholds.max_active_patterns`
   - Активных уроков ≤ `thresholds.max_active_lessons`
   - Сущности — без жёсткого лимита, но архивация без упоминаний >
     `thresholds.archive_entity_idle_months` мес.
4. **Ревизия:** раз в `thresholds.revision_interval_days` или по команде.
5. **Sync-check (v1.3):** автоматическая проверка согласованности
   порогов перед любой записью через `scripts/sync_check.py`.

## Уведомления пользователю

| Событие | Уведомлять? | Формат |
|---|---|---|
| Новый паттерн | Да | «Заметил паттерн: [...]» |
| Новый механизм | Да, если важный | «Зафиксировал механизм: [название]» |
| Новая ошибка | Да, всегда | «Заметил ошибку: [...] → механизм [...]» |
| Систематическая ошибка | Да, отдельно | «⚠ [N]-е повторение. Создан паттерн pat-NNN» |
| Обновление профиля | Да, явно | «Обновляю профиль: [...]» |
| Архивация старых | Нет | (тихо, в журнале) |
| Ревизия (v1.3: авто) | Да | «Запускаю плановую ревизию глобального слоя» |
| Пересборка графа | Нет | (тихо, в hook-логе) |
| Калибровка порогов (v1.3) | Да, с подтверждением | «Предлагаю изменить threshold X: был 3, стал 5. Подтвердить?» |

## Конфликт-резолвер

Что делать, если новое наблюдение **противоречит** существующему:

1. Записать противоречие в `14_audit_log.md` (тип
   `pattern_vs_pattern`, `mechanism_vs_rule`, и т.д.).
2. Уведомить: «Противоречие: [...]. Какое верно?»
3. После согласования — обновить старую запись и пометить
   `superseded_by_*`.
4. В v1.3: ссылка на противоречие автоматически появляется в
   `_generated/graph.html` как красное ребро.

## Чек-лист «прежде чем записать»

Перед фиксацией ЛЮБОЙ записи:

- [ ] Это сквозное (≥2 проектов или все диалоги)?
- [ ] Дубликата нет (grep)?
- [ ] Если паттерн — реально `thresholds.pattern`+ повторений?
- [ ] Категория и формат ID правильные (`pat-NNN`, `wm-NNN`, `ec-NNN`,
  `km-NNN`)?
- [ ] Связи с другими картами проставлены?
- [ ] Лимит активных не превышен?
- [ ] Уведомление пользователю готово (если требуется)?
- [ ] `00_index.md` обновлён?
- [ ] `sync_check.py` прошёл без ошибок (v1.3)?
