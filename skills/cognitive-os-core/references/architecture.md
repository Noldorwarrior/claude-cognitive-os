# Архитектура claude-cognitive-os v1.3.0-plugin

Детальная архитектура когнитивной OS и её место в экосистеме
пользователя.

## Четыре слоя когнитивной архитектуры

Вся память пользователя rakhman в работе с Claude разложена на четыре
уровня. Каждый слой отвечает за свой тип знаний и имеет свой скилл /
инструмент.

### Слой 1. Пользовательский

- **Где живёт:** system prompt Cowork / Claude Code
- **Что хранит:** 10 правил поведения (формат docx, триггер «ПРОМТ», язык
  общения, верификация, методологии)
- **Инструмент:** `user_preferences` (встроенный)
- **Характер:** стабильный, меняется редко, редактируется пользователем
  вручную через настройки

### Слой 2. Задачно-операционный

- **Где живёт:** `TASKS.md` в vault'е / рабочей папке
- **Что хранит:** текущие задачи, дедлайны, блокеры, назначения
- **Инструмент:** `productivity:memory-management`
- **Характер:** очень динамичный, обновляется несколько раз в день
- **Формат:** Markdown-чеклист с метаданными (owner, due, priority)

### Слой 3. Проектный

- **Где живёт:** `Projects/<name>/` — одна папка на проект
- **Что хранит:** знания внутри одного проекта (актеры, цели, решения,
  задачи, риски, ретро)
- **Инструмент:** `project-knowledge-maps` (отдельный скилл, если
  установлен; иначе — те же шаблоны, но без автозаполнения)
- **Характер:** среднединамичный, растёт вместе с проектом
- **Принцип lazy create:** карты создаются по мере необходимости, не все
  сразу

### Слой 4. Глобальный когнитивный

- **Где живёт:** `cognitive_os/` (один vault) или `Projects/<name>/`
  (если плагин используется как per-project solution)
- **Что хранит:** мета-знания, паттерны, профиль, уроки, сущности,
  механизмы, карты знаний
- **Инструмент:** `claude-cognitive-os` plugin (15 карт + 10 скиллов + 3
  субагента)
- **Характер:** низкодинамичный, но постоянно прирастает
- **Особенность:** работает поверх всего, не дублирует слои 2 и 3

## Принцип не-дублирования

Если знание касается **одного проекта** → оно живёт в слое 3
(`project-knowledge-maps`).

Если знание **сквозное** (через 2+ проекта или относится ко всем
диалогам в принципе) → оно живёт в слое 4 (этот плагин).

Примеры:
- «В проекте X ответственный за блок N — Иванов» → слой 3.
- «Пользователь предпочитает краткость, ≤500 слов» → слой 4.
- «Структура нацпроекта Молодёжь и дети — 8 ФП» → слой 4 (используется
  в 3+ проектах).
- «В проекте Y дедлайн защиты — 10.05.2026» → слой 2 (TASKS.md).

## Принцип не-инфляции

Чтобы глобальный слой не превратился в свалку, действуют ограничения.

### Паттерны (`02_patterns.md`)
- Не более `thresholds.max_active_patterns` активных (default 50).
- Старые — в `_archive/patterns_YYYY-MM.md`.
- Архивация через action-скилл `archive`.

### Уроки (`06_lessons_learned.md`)
- Не более `thresholds.max_active_lessons` активных (default 30).
- Старые сворачиваются в обобщённые принципы через
  `extract_principles.py` — при накоплении 3+ уроков с общим топиком
  извлекается принцип.

### Сущности (`07_global_entities.md`)
- Попадают только упомянутые в 2+ проектах.
- Порог управляется `thresholds.cluster_min_projects` (для сущностей
  по дефолту 2, для кластеров проектов — 3).

### Мета-решения (`04_meta_decisions.md`)
- Только архитектурные / стратегические / инструментальные.
- Не операционные (типа «на этой неделе делаем X»).

## Новое в v1.3: визуальный граф

Плагин строит **интерактивный HTML-граф** связей между всеми сущностями
vault'а: карты знаний, паттерны, механизмы, ошибки, уроки, проекты,
правила, люди, инструменты.

### Компоненты визуализации

- **`_generated/graph.html`** — интерактивный граф на vis-network (фильтры
  по типу сущности, ранжирование по частоте упоминаний, drill-down)
- **`_generated/backlinks.md`** — двусторонние обратные ссылки для
  AI-чтения (из какой карты ведёт ссылка, к какому якорю)
- **`_generated/graph.mermaid`** — fallback mermaid-граф для быстрого
  обзора в редакторах без vis-network

### Когда пересобирается

После каждого `Write` / `Edit` в папке проекта срабатывает **post-write
hook** с **debounce 3-5 секунд** и проверкой mtime — если за окно
debounce прошло несколько записей, хук запускается один раз по
последней. Это предотвращает race condition при быстрой серии правок.

Хук реализован в `hooks/hooks.json` плагина и вызывает:
1. `scripts/render_graph.py` — обновляет `_generated/graph.html`
2. `scripts/render_backlinks.py` — обновляет `_generated/backlinks.md`
3. `scripts/render_mermaid.py` — обновляет `_generated/graph.mermaid`

## Новое в v1.3: self-learning

Плагин умеет **подстраивать свои пороги** по реальным данным.

### Автокалибровка порогов (`calibrate_thresholds.py`)

Анализирует `11_confidence_scoring.md` и историю записей:
- Если 80%+ зафиксированных паттернов имели ≥5 повторений, а не 3 —
  предлагает поднять `thresholds.pattern` с 3 до 5.
- Если `02_patterns.md` переполнен (>80% от max_active) — предлагает
  снизить `max_active_patterns` или агрессивнее архивировать.
- Если уроки редко переходят в обобщённые принципы — предлагает
  снизить порог извлечения.

### Автоизвлечение принципов (`extract_principles.py`)

Сканирует `06_lessons_learned.md` и группирует уроки по топикам. Если
в одном топике накопилось 3+ уроков с похожим выводом, извлекает
общий принцип и добавляет его в `04_meta_decisions.md` с пометкой
`derived_from: [lesson-X, lesson-Y, lesson-Z]`.

### Sync-check (`sync_check.py`)

Проверяет согласованность порогов между `02_patterns.md`, `SKILL.md`,
`10_error_corrections.md`. Если где-то пороги разошлись — блокирует
любую запись до исправления.

## Новое в v1.3: 10 action-скиллов

Каждый — отдельный `skills/<name>/SKILL.md`:

| Скилл | Назначение |
|---|---|
| `init` | Инициализация cognitive-os в новой папке (создаёт CLAUDE.md + 11 шаблонов) |
| `migrate` | Миграция существующего проекта из произвольной структуры |
| `status` | Сводный отчёт: сколько карточек, паттернов, ошибок, размер графа |
| `graph` | Пересборка `_generated/graph.html` |
| `backlinks` | Пересборка `_generated/backlinks.md` |
| `reflect` | Self-reflection после комплексной задачи (интеграция с `govdoc-analytics:self-reflection`) |
| `calibrate` | Автокалибровка порогов |
| `archive` | Архивация старых сессий / паттернов |
| `patterns` | Обзор всех паттернов с ранжированием |
| `audit` | Аудит противоречий и скрытых допущений во всём vault'е |

## Новое в v1.3: 3 субагента

Для изоляции тяжёлых операций от основного контекста:

- **`cog-archivist`** (agents/cog-archivist.md) — архивация старых
  сессий, паттернов, сущностей в `_archive/YYYY-MM/`. Не трогает
  активные карты.
- **`cog-detector`** (agents/cog-detector.md) — детекция паттернов из
  истории сообщений и открытых карт. Запускается по триггеру `patterns
  detect` или автоматически раз в `thresholds.revision_interval_days`.
- **`cog-verifier`** (agents/cog-verifier.md) — применение 32 механизмов
  верификации из плагина `verification` к записям в `02_patterns.md`,
  `10_error_corrections.md`, `13_self_reflection.md`.

## Совместимость с Obsidian

Плагин совместим с Obsidian через:

1. **`[[wikilinks]]`** — все ссылки между картами используют wiki-формат,
   Obsidian их индексирует автоматически в Graph View.
2. **Frontmatter YAML** — все карты имеют YAML-шапку, совместимую с
   Obsidian properties.
3. **`.obsidian/` exclude** — папки `_generated/`, `.claude-plugin/`,
   `Claude/Plugins/` должны быть добавлены в exclude, чтобы не
   индексировать сгенерированные артефакты.
4. **Two-tier vault** — плагин работает и в режиме «один глобальный
   vault», и в режиме «sub-vault per project» (когда внутри проекта
   есть свой `.obsidian/`).

Подробности настройки — в `docs/integration.md` плагина и в разделе
«Настройки Obsidian» в README.

## Верификация П5 — связь с плагином verification

Сама сборка плагина и все существенные правки проходят через 32
механизма П5 (пресет «Максимум» плагина `verification`). Логи — в
`docs/verification_log.md`.

Для обычной работы плагина используется **пресет П13 «Аудитор»** —
когда проверяются чужие / внешние записи, и **пресет П14 «Итератор»**
— для многоитерационной работы с защитой от регрессии.
