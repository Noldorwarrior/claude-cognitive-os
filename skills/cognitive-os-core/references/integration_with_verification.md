# Интеграция cognitive-os-core со скиллами и плагинами

Этот документ описывает, как `cognitive-os-core` взаимодействует с другими
скиллами пользователя и плагинами экосистемы. Для каждой интеграции
указан **режим работы** и **fallback**, который применяется, если внешний
инструмент недоступен.

## Общая таблица интеграций

| Интеграция | Роль | Режим | Fallback |
|---|---|---|---|
| `productivity:memory-management` | Синхронизация с TASKS.md | Bidirectional (v1.3) | Read-only |
| `consolidate-memory` | Ревизия глобального слоя | Делегирование | `internal_minimal_revision` |
| `govdoc-analytics:self-reflection` | Самоаудит после задачи | Делегирование | `internal_8axis_scoring` |
| `verification` plugin (П1–П14) | Верификация записей | Делегирование | `manual_checklist` |
| `cognitive-os-graph` (этот плагин) | Визуализация | Внутренняя | — |
| `cognitive-os-calibration` (этот плагин) | Самокалибровка | Внутренняя | — |
| `govdoc-analytics:request-analytics` | Парсер запросов | Чтение | Ignore |
| `govdoc-analytics:speech-document-writer` | Создание текстов | Чтение паттернов | Ignore |
| `events-presentations:premium-presentations` | Презентации | Чтение паттернов | Ignore |
| `finance-data:spreadsheet-analyst` | Аналитика таблиц | Чтение паттернов | Ignore |

## 1. productivity:memory-management

**Роль:** управляет `TASKS.md` (слой 2 — задачно-операционный).

**Взаимодействие в v1.3 (bidirectional exchange):**

1. **cognitive-os-core → memory-management**:
   - Когда создаётся новая запись в `03_projects_registry.md`, core
     уведомляет memory-management о новом проекте, чтобы тот мог
     привязать задачи к нему.
   - Когда фиксируется паттерн про приоритизацию задач, core может
     предложить memory-management обновить категоризацию.

2. **memory-management → cognitive-os-core**:
   - При завершении задачи (статус `[x]` в TASKS.md) memory-management
     триггерит `reflect`-скилл для самоаудита.
   - При создании 3+ задач в одном проекте memory-management добавляет
     проект в `03_projects_registry.md`, если его там ещё нет.

**Точки синхронизации:**
- В начале каждой сессии: core читает `TASKS.md` для получения контекста
  текущих задач, memory-management читает `03_projects_registry.md` для
  понимания доступных проектов.
- При завершении задачи: обе системы обновляют свои артефакты
  синхронно.

**Fallback (если memory-management не установлен):**
- Core работает в read-only режиме с `TASKS.md`, если файл существует.
- Если файла нет — core игнорирует задачный слой и работает без него.

## 2. consolidate-memory

**Роль:** ревизия глобального слоя — объединение дубликатов, архивация
устаревших записей, проверка целостности.

**Режим работы:** когда пользователь запускает `пересмотри глобальный
слой` или action-скилл `calibrate` / `audit`, cognitive-os-core
**делегирует** работу consolidate-memory, передавая ему:
- Список активных карт
- Пороги из `thresholds` (max_active_patterns, max_active_lessons)
- Настройки архивации (archive_entity_idle_months)

consolidate-memory возвращает:
- Список предложений: что архивировать, что объединить, что удалить
- Новые пороги (если предлагает скорректировать)

Core применяет изменения после подтверждения пользователя.

**Fallback `internal_minimal_revision`:**

Если consolidate-memory не установлен, core выполняет минимальную
ревизию внутренне:

1. Скан `02_patterns.md` — если количество активных > `max_active_patterns`,
   находит паттерны с `last_used` > 6 месяцев и предлагает архивировать.
2. Скан `06_lessons_learned.md` — если > `max_active_lessons`, группирует
   уроки по топикам, предлагает свернуть в принципы.
3. Скан `07_global_entities.md` — находит сущности, упомянутые только в
   1 проекте, предлагает переместить обратно в project-level.
4. Проверка `00_index.md` — синхронизация счётчиков.

Этот fallback работает, но без оптимизаций consolidate-memory (например,
без semantic clustering паттернов).

## 3. govdoc-analytics:self-reflection

**Роль:** самоаудит Claude после комплексной задачи — скоринг по 8 осям,
извлечение уроков, журналирование решений.

**Режим работы:** при завершении задачи (по триггеру memory-management
или явному запросу) core **делегирует** самоаудит скиллу
govdoc-analytics:self-reflection. Тот возвращает:
- Скоринг 0–100% по 8 осям (корректность, полнота, формат, источники,
  логика, аудитория, калибровка, уроки)
- Журнал решений (что было решено и почему)
- Извлечённые уроки
- Предложения на улучшение

Core пишет результат в `13_self_reflection.md` и обновляет
`10_error_corrections.md`, если были обнаружены ошибки.

**Fallback `internal_8axis_scoring`:**

Если govdoc-analytics:self-reflection не установлен, core выполняет
самоаудит внутренне по 8 осям, используя упрощённую эвристику:

| Ось | Упрощённая проверка |
|---|---|
| Корректность | Все ли цифры / даты из запроса перенесены? |
| Полнота | Все ли пункты запроса выполнены? |
| Формат | Соответствует ли формат пользовательским правилам? |
| Источники | Указаны ли источники для фактов? |
| Логика | Есть ли противоречия в ответе? |
| Аудитория | Соответствует ли глубина уровню аудитории? |
| Калибровка | Уверенность соответствует точности? |
| Уроки | Что извлечено для следующего раза? |

Результат — менее глубокий, чем у полноценного self-reflection, но
достаточный для базовой фиксации.

## 4. verification plugin (П1–П14)

**Роль:** верификация записей в картах знаний через 32 механизма
верификации, сгруппированные в 14 пресетов.

**Режим работы:** при фиксации существенных записей (новый паттерн, новая
ошибка, новое мета-решение) core **делегирует** верификацию плагину
`verification`. По умолчанию:

- **Новый паттерн** → пресет **П13 «Аудитор»** (проверка чужих записей,
  так как паттерн обобщает наблюдения Claude о поведении пользователя)
- **Новая ошибка в `10_error_corrections.md`** → пресет **П5 «Максимум»**
  (критический документ, все 32 механизма)
- **Новое мета-решение** → пресет **П12 «Стратег»**
- **Итеративная правка карты** → пресет **П14 «Итератор»** (защита от
  регрессии и дрейфа смысла)
- **Обновление `13_self_reflection.md`** → пресет **П1 «Аналитик»**

**Hook на завершение задачи:**

В v1.3 верификационный плагин подписан на событие «задача завершена»
(from memory-management / self-reflection) и автоматически запускает
верификацию:
- Если в ходе задачи были правки в картах → П14.
- Если создавался документ → пресет под тип документа (П1, П2, П7, П12).
- Результат записывается в `10_error_corrections.md` (если нашли
  расхождения) и `13_self_reflection.md` (общий лог).

**Fallback `manual_checklist`:**

Если `verification` plugin не установлен, core применяет ручной чек-лист
перед каждой записью (см. «Чек-лист перед каждой записью» в SKILL.md):

- [ ] Это действительно сквозное знание?
- [ ] Это уже не зафиксировано (grep)?
- [ ] Паттерн — действительно N+ повторений?
- [ ] Есть краткая формулировка ≤ 200 символов?
- [ ] ID присвоен по формату?
- [ ] `00_index.md` обновлён?
- [ ] Пользователь уведомлён?

Ручной чек-лист не заменяет 32 механизмов, но обеспечивает минимальный
уровень качества.

## 5. cognitive-os-graph (внутри плагина)

**Роль:** визуализация связей между картами, паттернами, проектами,
сущностями.

**Режим работы:** core обновляет данные в картах → `cognitive-os-graph`
через post-write hook пересобирает `_generated/graph.html`,
`_generated/backlinks.md`, `_generated/graph.mermaid`.

**Обратный поток:** при запросе «покажи граф» core делегирует открытие
HTML-графа скиллу `cognitive-os-graph`. Тот также отвечает за
drill-down: клик по узлу в графе → открытие соответствующей карты.

**Нет fallback** — это часть плагина, всегда доступна.

## 6. cognitive-os-calibration (внутри плагина)

**Роль:** автокалибровка порогов, извлечение принципов, sync-check.

**Режим работы:** запускается action-скиллом `calibrate` или
автоматически раз в `thresholds.revision_interval_days`. Анализирует:
- `11_confidence_scoring.md` — фактические показатели уверенности
- `02_patterns.md` — фактические кол-ва повторений паттернов
- `06_lessons_learned.md` — группировки уроков по топикам

Возвращает:
- Предложения по корректировке порогов (`thresholds.pattern` с 3 на 5,
  `max_active_patterns` с 50 на 40 и т.д.)
- Новые принципы из уроков (записываются в `04_meta_decisions.md`)
- Alerts о десинхронизации порогов между файлами

**Нет fallback** — часть плагина.

## 7. govdoc-analytics:request-analytics

**Роль:** парсер входящих запросов — определяет тип задачи, домен,
уровень сложности, необходимые скиллы.

**Режим работы:** core **читает** результаты request-analytics, чтобы
выбрать стартовый уровень reading ladder:
- Если request-analytics определил «простой вопрос» → уровень 0.
- Если «методологический выбор» → уровень 2.
- Если «содержательная задача в домене X» → уровень 3 с фокусом на `km-X`.

**Fallback:** если request-analytics не установлен, core использует
эвристики из `reading_ladder.md` напрямую по ключевым словам запроса.

## 8. govdoc-analytics:speech-document-writer

**Роль:** создание официальных текстов (речи, письма, записки, справки,
концепции).

**Режим работы:** core **передаёт** speech-document-writer релевантные
паттерны из `02_patterns.md`, мета-решения из `04_meta_decisions.md` и
предметные знания из `08_knowledge_maps.md` для применения в тексте.

Если speech-document-writer создаёт новый документ, core фиксирует:
- Новый тип документа → возможно, новый паттерн в `02_patterns.md`
- Новые термины → `05_global_glossary.md`
- Новые сущности (если упомянуты 2+ раза) → `07_global_entities.md`

**Fallback:** без speech-document-writer core просто не предоставляет
специализированных шаблонов. Тексты создаются generic-подходом.

## 9. events-presentations:premium-presentations

**Роль:** создание презентаций (pptx + HTML) по правилу пользователя #7.

**Режим работы:** core **передаёт** паттерны про визуализацию,
предпочтения по дизайну, домен презентации. premium-presentations
применяет их при создании.

Результат завершения презентации → триггерит self-reflection → запись в
`13_self_reflection.md`.

**Fallback:** без premium-presentations используется стандартный pptx
скилл без интерактивной HTML-версии.

## 10. finance-data:spreadsheet-analyst

**Роль:** анализ табличных данных через 53 протокола.

**Режим работы:** core **передаёт** паттерны про выбор протоколов
пользователем (например, «для плана-факта — pat-023»), домен данных.

Выбор нового протокола → возможно, новый паттерн в `02_patterns.md`.

**Fallback:** без spreadsheet-analyst используются базовые скиллы
`finance-data:analyze`.

## 11. Связь с верификацией П5 (32 механизма)

Сама сборка плагина `claude-cognitive-os` и все существенные правки
проходят через **пресет П5 «Максимум»** плагина `verification` — все 32
механизма верификации.

**Что проверяется:**
- Корректность переноса из v1.0.1 (М1: точный перенос цифр/дат/имён)
- Согласованность порогов между `SKILL.md::thresholds`, шаблонами карт
  и действиями скриптов (М22: согласованность файлов)
- Отсутствие противоречий между картами (М7: поиск противоречий)
- Полнота покрытия case'ов из v1.0.1 (М15: полнота)
- Защита от регрессии при правках (М25: защита от регрессии)
- Целостность ссылок во всех картах и references (М32: ссылочная целостность)

**Где хранятся логи:**
- `docs/verification_log.md` в корне плагина — отчёты по каждому
  существенному этапу (Stage 1, Stage 2, ... Stage 10 — финальная П5)
- `13_self_reflection.md` в vault'е пользователя — применение пресета
  П14 при каждой итеративной правке карты

**Пресеты для повседневной работы:**

| Ситуация | Пресет | Объяснение |
|---|---|---|
| Новая ошибка | П5 «Максимум» | Критически важная запись |
| Новый паттерн | П13 «Аудитор» | Обобщение наблюдений |
| Мета-решение | П12 «Стратег» | Архитектурное решение |
| Правка карты | П14 «Итератор» | Защита от регрессии |
| Перевод ошибки в принцип | П5 + П14 | Критическая итеративная |

## 12. Интеграция с субагентами плагина

### cog-archivist
- **Когда активируется:** action-скилл `archive` или автоматически при
  превышении `max_active_*`.
- **Что делает:** переносит старые паттерны / сессии / сущности в
  `_archive/YYYY-MM/`, не трогая активные.
- **Связь с core:** core вызывает субагента при необходимости, передаёт
  список кандидатов на архивацию.

### cog-detector
- **Когда активируется:** триггер `patterns detect`, action-скилл
  `patterns`, или автоматически раз в `thresholds.revision_interval_days`.
- **Что делает:** сканирует историю сообщений и карты, ищет новые
  паттерны, превысившие порог `thresholds.pattern`.
- **Связь с core:** возвращает списком предложенные паттерны, core
  применяет после подтверждения пользователя.

### cog-verifier
- **Когда активируется:** перед записью паттерна / ошибки / мета-решения.
- **Что делает:** применяет 32 механизма верификации (П5) к записи.
- **Связь с core:** core блокирует запись, пока не получил зелёный свет
  от cog-verifier. Если fallback `manual_checklist` — блокировка снимается
  после прохождения чек-листа.
