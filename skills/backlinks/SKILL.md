# backlinks — Обратные ссылки

## Назначение

Для каждого ID в воркспейсе собрать **обратные ссылки**: какие записи
(в каких карточках) на него ссылаются. Обсидиан строит это
автоматически во внутреннем индексе, но мы сохраняем текстовый artefact
для:
- Просмотра вне Обсидиана.
- Машинного чтения (скрипты, субагенты).
- Аудита (никаких потерянных связей).

## Когда запускается

- Явная команда (`покажи backlinks`, `обратные ссылки`).
- Post-write hook после изменений в любой карточке (с debounce 3-5 сек).
- После [[init]] / [[migrate]] — обязательно.
- При подозрении на разорванные ссылки — через [[audit]].

## Когда НЕ запускается

- Воркспейс пустой.
- Hook сработал слишком часто (debounce блокирует).

## Workflow

### 1. Сбор всех ID в воркспейсе

Regex-скан по всем `.md` файлам. **Все паттерны обрамлены word-boundary
`\b`**, чтобы избежать ложных срабатываний внутри слов (например,
`impact-001-plan` не должен совпадать с `pat-001`):

```
\bpat-\d{3}\b    → patterns
\bwm-\d{3}\b     → mechanisms
\bec-\d{3}\b     → errors
\bkm-\d{3}\b     → knowledge maps
\bmd-\d{3}\b     → meta decisions
\bterm-\d{3}\b   → glossary terms
\bent-\d{3}\b    → entities
\bproj-\d{3}\b   → projects
\blesson-\d{3}\b → lessons
\bsr-\d{3}\b     → self-reflections
\baudit-\d{3}\b  → audit log entries
\bdomain-\d{3}\b → confidence domains
\bcluster-\d{3}\b → project clusters
```

Также собираются имена карточек вида `[[00_index]]`, `[[01_user_profile]]`,
... — по точному совпадению внутри `[[...]]`.

### 2. Парсинг ссылок

Для каждого `.md`:
- Находим все `[[target]]` и `[[target#anchor]]`.
- Регистрируем ссылку: (source_file, target_id, anchor, строка).

### 3. Построение индекса backlinks

```python
backlinks = {
  "pat-001": [
    {"from": "09_working_mechanisms.md", "anchor": "wm-003", "line": 42},
    {"from": "13_self_reflection.md", "anchor": "sr-005", "line": 18},
  ],
  "ent-020": [ ... ],
  ...
}
```

### 4. Рендер backlinks.md

```markdown
# Обратные ссылки — {{DATE}}

## Паттерны

### pat-001 — Короткое название
Ссылаются:
- [[09_working_mechanisms#wm-003]]
- [[13_self_reflection#sr-005]]

### pat-002 — ...
...

## Механизмы

### wm-001 — ...
Ссылаются:
- [[10_error_corrections#ec-005]]
- [[04_meta_decisions#md-002]]

## Ошибки

...

## Сущности

### ent-020 — РОЗ
Ссылаются:
- [[03_projects_registry#proj-001]]
- [[03_projects_registry#proj-005]]
- [[03_projects_registry#proj-012]]
- [[08_knowledge_maps#km-002]]
...

## Потерянные ссылки

Если есть `[[pat-999]]`, которого не существует:

### [[pat-999]]
Упоминается в:
- [[09_working_mechanisms#wm-007]] (строка 24) — **ВИСЯЧАЯ ССЫЛКА**

→ Зафиксирована в [[14_audit_log#audit-007]] как `regression`.
```

### 5. Сравнение с предыдущей версией (diff)

Если `backlinks.md` уже существует:
- Посчитать diff: какие связи появились, какие пропали.
- Пропавшие ссылки → проверить, не было ли удаления без замены (М25
  защита от регрессии).

### 6. Валидация

- Каждый ID имеет секцию (даже если backlinks нет — пометить «Нет
  обратных ссылок» → кандидат на архивацию).
- Нет дубликатов ссылок из одного места.

### 7. Запись и уведомление

- `_generated/backlinks.md` — перезапись.
- Уведомление: количество ID, количество ссылок, количество висячих.

## Использование backlinks

### В Обсидиане

Backlinks-панель уже работает автоматически. Текстовый файл — дубляж
для внешних инструментов.

### В субагентах

- [[cog-detector]] — использует backlinks для обнаружения изолированных
  узлов (кандидаты на архив).
- [[cog-verifier]] — сверяет ссылки при проверке П14.

### В скриптах

- `sync_check.py` — проверяет, что изменения в одной карточке
  отражены в связанных.
- `audit_contradictions.py` — находит циклические зависимости.

### В аудите

- [[audit]] — использует как источник для поиска противоречий.

## Режимы

### `backlinks` (default)

Полный обзор всех ID.

### `backlinks --id=<id>`

Только для одного ID (например, `--id=ent-020`).

### `backlinks --orphans`

Только ID без входящих ссылок (кандидаты на архив).

### `backlinks --broken`

Только висячие ссылки (→ в [[14_audit_log]]).

## Команды

| Команда | Действие |
|---|---|
| `покажи backlinks` | Полный рендер |
| `обратные ссылки` | Алиас |
| `кто ссылается на X` | `--id=X` |
| `изолированные узлы` | `--orphans` |
| `висячие ссылки` | `--broken` |

## Интеграции

- [[graph]] — парный скилл, использует тот же источник.
- [[cog-detector]] — читает `backlinks.md` для обнаружения изолятов.
- [[audit]] — источник для поиска противоречий.
- `hooks/hooks.json` — post-write на любую карточку → этот скилл
  (debounce 3 сек).
- Obsidian Backlinks plugin — не заменяет, а дополняет.

## Производительность

- Линейно от размера воркспейса.
- Скан 2000 ID / 100 карточек: ~2 сек.
- Кеширование mtime — если карточки не менялись с последнего запуска,
  skip.

## Верификация

**П13 «Аудитор»** — после рендера:
- №22 согласованность файлов.
- №32 ссылочная целостность (нет висячих).

Если найдены висячие — автоматически создаётся запись в
[[14_audit_log]] с типом `regression`.

## Частые ошибки

1. **Висячие ссылки не зафиксированы** → тихие потери связности.
   Защита: шаг 4 + auto-audit.
2. **Дубликаты ссылок** из одного места (например, ссылка в заголовке
   и в теле раздела) → засоряют отчёт. Защита: дедупликация по (from, target).
3. **Anchor не совпадает с реальным якорем** (опечатка в ID) → воспринимается
   как висячая. Защита: fuzzy-match подсказка.
