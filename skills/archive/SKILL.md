# archive — Архивация устаревших записей

## Назначение

Переносить записи, которые:
- Не использовались > N месяцев.
- Помечены пользователем как `archived`.
- Вытеснены новыми (superseded_by).

…из активных карточек в `archive/<карточка>/` с сохранением ID и ссылок.
Это не удаление — запись остаётся доступной, просто не загромождает
активный пул.

## Когда запускается

- Явная команда «архивируй», «очисти устаревшее».
- Автоматически раз в `thresholds.archive_interval_days` (default 90).
- После калибровки, если `max_active_*` пороги превышены.
- Post-hook после закрытия ошибки (`ec-*.status = closed_with_protection`
  + last_seen > 12 месяцев).

## Когда НЕ запускается

- Воркспейс пустой.
- Запись имеет активные входящие ссылки (backlinks > 0) — тогда
  предупредить, не архивировать.
- Критичные карточки (00, 01) — никогда не архивируются.

## Workflow

### 1. Сбор кандидатов на архив

Проход по активным карточкам с критериями:

| Карточка | Критерий архива |
|---|---|
| `02_patterns` | `last_seen > archive_pattern_idle_months` (6) |
| `06_lessons_learned` | `last_used > 12 месяцев` ИЛИ `max_active_lessons` превышен |
| `07_global_entities` | `last_used > archive_entity_idle_months` (6) |
| `09_working_mechanisms` | `replaced_by_wm` указан + 3 месяца прошло |
| `10_error_corrections` | `status = closed_with_protection` + `last_seen > 12 месяцев` |
| `13_self_reflection` | Возраст > 24 месяца |
| `03_projects_registry` | `status = finished/archived` + 6 месяцев прошло |

### 2. Проверка входящих ссылок

Для каждого кандидата:
- Читаем `_generated/backlinks.md` или пересобираем.
- Если `backlinks_count > 0` в активных карточках → предупреждение.
- Если `backlinks_count == 0` — безопасно архивировать.
- Если ссылки только из уже архивированных — безопасно.

### 3. Создание структуры архива

```
archive/
├── patterns/
│   └── pat-007.md    (отдельный файл, сохраняющий всё содержимое)
├── lessons/
├── entities/
├── mechanisms/
├── errors/
├── reflections/
└── projects/
```

Каждая архивированная запись — **отдельный `.md` файл** с сохранением
frontmatter, но с добавленным:

```yaml
archived: true
archived_date: "2026-04-20"
archived_reason: "not_used_6_months"
original_card: "02_patterns"
```

### 4. Замена ссылок

Ссылки `[[pat-007]]` из активных карточек:
- **Не ломаются** — Обсидиан их резолвит через weights или по имени файла
  (мы сохраняем имя).
- Для надёжности: в момент архивации заменяем на
  `[[pat-007|pat-007 (архив)]]` в активных карточках.

### 5. Удаление из активной карточки

Удаляем запись из активного `.md` (например, из `02_patterns.md`):
- Сохраняем целостность YAML (счётчики).
- Добавляем в раздел `## Архивированные (ссылки)` краткий пойнтер:

```markdown
## Архивированные

- [[archive/patterns/pat-007]] — «pre-mortem в конце» (архив: 2026-04-20,
  причина: not_used_6_months).
```

### 6. Обновление счётчиков

- `active_count` в карточке: −1.
- `archived_count`: +1.
- `00_index.md` синхронизирован.

### 7. Rebuild `_generated/`

После архивации:
- [[graph]] — перерисовка графа (архивные узлы в отдельном слое или
  скрыты).
- [[backlinks]] — пересбор.

### 8. Аудит

Если архивация прошла с предупреждениями (были входящие ссылки) —
запись в [[14_audit_log]] как `scope_drift` или `regression`.

### 9. Уведомление

```
Архивация завершена.
Перенесено:
- patterns: 3 → archive/patterns/
- lessons: 5 → archive/lessons/
- entities: 2 → archive/entities/
Активных записей: было 89, стало 79.
Следующая автоархивация: 2026-07-19.
```

## Восстановление из архива

Команда «восстанови pat-007»:

1. Читаем `archive/patterns/pat-007.md`.
2. Удаляем `archived: true`, `archived_date`, `archived_reason`.
3. Обновляем `last_seen: {{DATE}}`.
4. Копируем запись обратно в активную карточку.
5. Удаляем запись из `archive/`.
6. Перестраиваем ссылки.

## Режимы

### `archive` (default)

Автоматическая архивация по критериям.

### `archive --dry-run`

Показать кандидатов, не применять.

### `archive --type=<type>`

Только один тип (`patterns`, `entities`, ...).

### `archive --id=<id>`

Принудительная архивация одной записи.

### `restore --id=<id>`

Восстановление из архива.

## Верификация

**П14 «Итератор»** — обязательно для архивации:
- №25 защита от регрессии (ничего не потеряно).
- №26 дрейф смысла (ссылки не поменяли значение).
- №24 diff было/стало.
- №22 согласованность файлов.
- №32 ссылочная целостность.

Запись в `docs/verification_log.md`.

## Частые ошибки

1. **Архивация с входящими ссылками** — ссылки становятся висячими.
   Защита: шаг 2 + предупреждение.
2. **Архивация критической карточки** (00, 01) → каскад сломанных ссылок.
   Защита: чёрный список в шаге 1.
3. **Не обновлены счётчики** → `00_index` врёт. Защита: шаг 6 обязателен.
4. **Потеря frontmatter** при переносе → запись теряет метаданные.
   Защита: копирование целиком, не парсинг-переписывание.

## Rollback

Если архивация прошла с ошибкой:
- Восстановление из `cognitive-os-backup-{{DATE}}/` (если был бэкап).
- Или индивидуальное `restore --id=X` для каждой.
