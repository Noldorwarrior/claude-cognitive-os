# audit — Глубокий аудит воркспейса

## Назначение

Найти и зафиксировать все **расхождения, противоречия, регрессии, дрейфы,
рассинхронизации** в когнитивном воркспейсе:

- Противоречия между картами знаний (km-NNN vs km-MMM).
- Конфликты между паттернами (pat-NNN vs pat-MMM).
- Противоречия между проектами (proj-NNN vs proj-MMM).
- Рассинхрон порогов (thresholds в SKILL.md vs в карточках).
- Регрессии (М25) — что-то пропало без причины.
- Дрейф смысла (М26) — определение сместилось между версиями.

Результат — записи `audit-NNN` в [[14_audit_log]] с предложениями
разрешения.

## Когда запускается

- Явная команда («проведи аудит», «проверь противоречия»).
- Автоматически раз в `thresholds.revision_interval_days` (default 30).
- После крупных изменений (10+ новых записей в любой карточке).
- По триггеру от [[cog-verifier]] при обнаружении подозрения.
- Перед релизом / крупной публикацией.

## Когда НЕ запускается

- Воркспейс пустой.
- Аудит уже был < 7 дней назад (кроме принудительного режима).
- Активная транзакция записи в карточки.

## Workflow

### 1. Проверка синхронизации (sync_check)

Запуск `sync_check.py`:
- Пороги в `SKILL.md` соответствуют порогам в карточках?
- `00_index.counters` совпадают с фактическими записями?
- Frontmatter YAML валиден?

Расхождения → `audit-NNN` типа `threshold_mismatch`.

### 2. Проверка висячих ссылок

Запуск [[backlinks]] в режиме `--broken`.

Для каждой висячей `[[target]]`:
- Что ожидалось?
- Возможно, target был переименован?
- Возможно, удалён без замены?

→ `audit-NNN` типа `regression`.

### 3. Парный поиск противоречий (knowledge_conflict)

Для пар `[km-A, km-B]` в [[08_knowledge_maps]]:
- Семантическая близость (общая тема).
- Выявление противоречивых утверждений:
  - «8 ФП» vs «7 ФП».
  - «Централизованный» vs «децентрализованный».
  - Другие бинарные / числовые несоответствия.

Делегирование в `audit_contradictions.py` + [[cog-verifier]] с
пресетом **П13**.

→ `audit-NNN` типа `knowledge_conflict`, severity по степени расхождения.

### 4. Конфликты паттернов (pattern_conflict)

Для пар `[pat-A, pat-B]` в [[02_patterns]]:
- Несовместимые триггеры (оба срабатывают на одно, но предписывают разное).
- Противоположные предписания.

→ `audit-NNN` типа `pattern_conflict`.

### 5. Конфликты проектов (project_conflict)

Для пар `[proj-A, proj-B]` в [[03_projects_registry]]:
- Общий домен + противоположные цели.
- Общие сущности + противоречивые роли.

→ `audit-NNN` типа `project_conflict`, отметка в [[12_cross_project_graph]]
как conflict-edge.

### 6. Дрейф смысла (scope_drift)

Для каждой часто-итерируемой карточки (есть version history):
- Сравнить определения ключевых терминов между v1 и current.
- Выявить неявные переопределения.

→ `audit-NNN` типа `scope_drift`, механизм М26.

### 7. Регрессии (regression)

Для изменений за последние N дней (`revision_interval_days`):
- Что было в предыдущей версии, чего нет сейчас?
- Удаление с обоснованием или без?

→ `audit-NNN` типа `regression`, механизм М25.

### 8. Создание записей audit-NNN

Каждое расхождение → запись в [[14_audit_log]] по шаблону:
- Тип, severity, источники, суть, гипотеза причины, предложенное
  разрешение.
- Критичные (high / critical) — автоматически запускают **П5 «Максимум»**
  через [[cog-verifier]].

### 9. Диалог с пользователем

Через `AskUserQuestion`:

```
Аудит обнаружил 7 расхождений:
- 3 высокой severity (критичные).
- 2 средних.
- 2 низких.

Что делать:
A. Разобрать сейчас критичные (3 расхождения).
B. Разобрать все по очереди.
C. Сохранить в журнал, разобрать позже.
D. Показать подробности каждого.
```

### 10. Применение разрешений

Для каждого одобренного разрешения:
- Применить изменение в соответствующей карточке.
- Обновить статус audit-NNN на `resolved` или `accepted_as_context_diff`.
- Верификация **П14 «Итератор»** после правки.

### 11. Отчёт

Сохранить в `docs/audits/audit_report_{{DATE}}.md`:
- Что найдено, что разрешено, что открыто.
- Тренд: больше / меньше / столько же расхождений, чем в прошлый аудит.
- Рекомендации.

### 12. Rebuild `_generated/`

После разрешений:
- [[graph]] — conflict-edges обновляются.
- [[backlinks]] — пересобираются.

## Режимы

### `audit` (default)

Полный аудит всех 7 проверок.

### `audit --contradictions-only`

Только противоречия (шаги 3, 4, 5).

### `audit --sync-only`

Только проверка синхронизации (шаг 1).

### `audit --broken-links-only`

Только висячие ссылки (шаг 2).

### `audit --since=<date>`

Только изменения после указанной даты.

### `audit --dry-run`

Найти, но не создавать записи.

## Команды

| Команда | Действие |
|---|---|
| `проведи аудит` | Полный |
| `проверь противоречия` | `--contradictions-only` |
| `проверь воркспейс` | Полный |
| `найди расхождения` | Полный |
| `висячие ссылки` | `--broken-links-only` |
| `аудит с 2026-04-01` | `--since=2026-04-01` |
| `покажи план аудита` | `--dry-run` |

## Интеграции

- [[cog-verifier]] (subagent) — П5 / П13 для разборов.
- [[14_audit_log]] — основная карточка.
- [[graph]], [[backlinks]] — источники и потребители.
- `scripts/audit_contradictions.py` — исполнитель шагов 3-5.
- `scripts/sync_check.py` — исполнитель шага 1.
- [[verification]] plugin — все пресеты.
- [[calibrate]] — может вызываться после аудита, если накопились
  systemic-ошибки.

## Верификация

- **Сам аудит:** **П13 «Аудитор»** — это его профиль.
- **Критичные находки:** эскалация в **П5 «Максимум»** через cog-verifier.
- **После разрешений:** **П14 «Итератор»** — защита от регрессии.

## Частые ошибки

1. **Ложные противоречия** (на самом деле разные контексты). Защита:
   предложение `accepted_as_context_diff` как валидный исход.
2. **Разрешения без одобрения пользователя** → молчаливые правки.
   Защита: шаг 9 обязателен.
3. **Каскадные противоречия** (одно порождает много) → взрыв записей.
   Защита: группировка и поиск корня.
4. **Пропуск висячих ссылок** (фокус только на контенте). Защита: шаг 2
   всегда первым.

## Rollback

Если разрешения применены и это оказалось неверно:
- Восстановление из git-истории воркспейса (если подключён git).
- Или из `cognitive-os-backup-{{DATE}}/`.
